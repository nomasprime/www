Most conversations about AI agents start with the model. Which one is smarter? Which one is faster? Which one writes better code?

Those questions matter, but they miss where a lot of agent quality actually comes from. A useful software engineering agent is not just an LLM. It is a system around an LLM: instructions, context, tools, validation, runtime behaviour, and feedback loops all shaping what the model can do.

Before we can integrate agents well, we need to understand that stack.

## What Is AI?

There’s a lot of hype and misunderstanding around AI. Sure, the technologies are impressive, but productive people aren’t interested in hype, they want to understand how to leverage technology to increase their effectiveness.

Let's start with a practical definition of AI:

AI is the ability of computational systems to perform tasks usually associated with human intelligence, such as learning, reasoning, problem-solving, perception, decision-making, and taking actions toward defined goals.[^1]

## When Does AI Become Agentic?

This is similar to asking when a process becomes automated. The key difference is control flow. In traditional automation, the programmer defines which step runs next. In agentic AI, the model observes the current situation and helps decide what should happen next at runtime.

That does not mean the whole system is implicit. The agentic stack is a hybrid: explicit scaffolding around an implicit reasoning core. Instructions, tools, validation, permissions, adapters, and control loops constrain and direct the model’s behaviour so its runtime decisions remain useful.

So the answer is similar: AI becomes agentic when it can pursue a goal through a feedback loop. It chooses an action, observes the result, and adjusts what it does next without every step being explicitly predetermined. From that point onwards, agency exists on a continuum.

For example, ChatGPT is an AI assistant with agentic capabilities, such as searching the web or carrying out multi-step tasks using tools.

By contrast, Codex is a coding agent. Its primary purpose is to pursue software engineering goals by planning, using development tools, observing the results, and iterating until the task is complete.

Both are agentic. The difference is one of degree.

So, it’s often more useful to think in terms of degrees of agency rather than whether something “is” or “isn’t” an agent.

## What Does An Agentic AI Stack Consist Of?

An agentic AI is a system with an LLM as a core component. The runtime is the execution layer of the agentic system: it assembles instructions and context, exposes allowed tools, runs a control loop, enforces validation and permissions, and calls the model through an adapter. The loop continues until the agent's purpose is fulfilled or the agent is unable to continue.

### Instructions

These define the agent's purpose and come from multiple sources, eg:

- System prompt
- Developer prompt
- User prompt
- AGENTS.md
- Skills
- Project rules
- Company policies
- Style guides
- Safety policies

For a coding agent, instructions might say:

- Run tests before finishing
- Use pnpm, not npm
- Do not edit generated files
- Follow this repo’s naming conventions
- Ask before making destructive changes

### Context

The given context should be all the relevant information to complete the instructions, eg:

- Conversation history
- The user’s latest message
- Relevant files
- Search results
- Tool outputs
- Code snippets
- Error logs
- Repository structure
- Retrieved documentation
- Memory

Instructions shape behaviour, context supplies situational knowledge.

A lot of AI application quality comes from choosing the right context. Too little context and the model guesses. Too much irrelevant context and the model gets distracted.

### Tools

These are external capabilities exposed through the agent runtime. They're integrations that allow the agent to interact with other systems to get things done, eg:

- Read a file
- Write a file
- Search the web
- Run Python
- Run terminal commands
- Query a database
- Call an API
- Send an email
- Read a calendar
- Generate an image
- Run tests
- Use Git
- Compile code

The runtime decides how tools are exposed and when they may be called.

For example, the model infers it should run the tests, and the runtime uses the terminal tool to run the relevant test command.

### Control Loop

The control loop is what lets the system move from a single model response to goal-directed behaviour. It gives the agent a repeated cycle for choosing an action, observing the result, and deciding whether to continue, stop, or ask for help:

1. Think
2. Act
3. Observe
4. Repeat from 1 unless finished, blocked, or waiting for input

For example, when told to fix failing tests:

1. Inspect repo
2. Read AGENTS.md
3. Ask model for likely cause
4. Edit file
5. Run tests
6. Read error
7. Edit again
8. Run tests again
9. Summarise changes

### Validation

Validation is how the runtime checks whether an action or output is acceptable before the loop continues.

Some checks are advisory. The runtime might ask the model to revise an answer that is too vague, incomplete, or incorrectly formatted.

Other checks are enforced. The runtime might block a tool call, require user approval, reject invalid JSON, stop before a destructive action, or fail the task if tests do not pass.

Examples:

- Is this tool call allowed?
- Is user approval required before continuing?
- Does the output match the required JSON schema?
- Did the tests pass?
- Did the code compile?
- Is the model hallucinating a file path?
- Does this violate a company policy?

In coding agents, validation is often concrete:

- Run tests
- Run linter
- Check diff
- Check type errors

### LLM Adapter

The model adapter is the translation layer between the runtime and a specific LLM provider/model.

Different models have different APIs, formats, strengths, and quirks.

The adapter handles things like:

- Message formatting
- Tool-call schema formatting
- Token limits
- Streaming responses
- Retry behaviour
- Model-specific parameters
- JSON mode
- Function calling
- Vision inputs
- File inputs
- Error handling

For example, your runtime might have a generic internal request:

Ask model:
- Here are the instructions.
- Here is the context.
- These tools are available.
- Return either an answer or a tool call.

The adapter converts that into the exact format required by:

- OpenAI
- Anthropic
- Google
- local Llama model
- some future model

This is what makes model swapping possible, though not always painless.

### LLM

The LLM is the language model itself.

It is the part that takes input and generates output.

It can:

- Interpret language
- Generate text
- Write code
- Summarise
- Classify
- Plan
- Infer intent
- Suggest tool calls
- Produce structured data

## Summary

The agentic AI stack gives us a practical way to reason about agent quality. It makes agentic AI manageable by putting explicit engineering scaffolding around an implicit reasoning core.

If an agent produces poor results, the model might be the problem, but it is not the only place to look. The issue might be unclear instructions, missing context, weak tools, loose validation, a brittle runtime, or a feedback loop that does not surface the right information at the right time.

That is why integrating agentic AI is not just model selection. It is systems design. The work is to shape the stack around the model so the agent can understand the goal, act safely, adjust based on feedback, and produce useful results with fewer corrections.

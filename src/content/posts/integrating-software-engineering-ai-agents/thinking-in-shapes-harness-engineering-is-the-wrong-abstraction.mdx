I've come to think about AI, especially an LLM, as a shape-producing machine: like a giant ball of play dough continuously pushed through stencils until you end up with a chunk of play dough that satisfies.

In a transformer, roughly speaking, we can start to see how the play dough (tokens) gets reshaped through the various layers (stencils) to produce output:

1. Data is broken down into small units called tokens
2. Each token is assigned an ID and mapped to an embedding vector
3. Each token-position is repeatedly transformed through layers into new hidden-state vectors
4. Those hidden states become increasingly context-shaped
5. The final hidden states are used to produce probabilities over possible outputs

LLMs can produce a vast number of permutations. The bigger the LLM, the larger the number of shapes it's capable of producing.

## Harnesses Narrow The Shape Space

Building an agent harness is really about narrowing or reducing those shape permutations. It is a constraint system that carves out an admissible subset from an ambient shape space.

Thinking about it like this makes it easier to reason about the tangible benefits instead of getting distracted by hype.

Each layer reduces the number of possible shapes by adding more specific context and constraints:

1. LLM
    1. Data, training, and alignment
    2. Guardrails
2. Harness
    1. Project files and agent-specific guidance
    2. Tools
    3. Prompting
    4. Guardrails

While we might not have control over an individual LLM, it's important to think of it as a harness dependency. If the LLM changes, the harness is affected to varying degrees.

The trick to maximising value is adjusting the context at each layer, from general to more specific. Done well, the agent can execute effectively across the widest range of use cases within its purview and produce admissible shapes in the fewest iterations.

That framing is useful, but it also gives the harness too much credit.

## Software Engineering Was Already Shapes

Agents make the shaping problem more visible, but they did not create it. Software engineering has always been about constraining, evaluating, evolving, and producing admissible shapes.

Software engineering is full of constraints that shape our output:

- Languages shape what can be expressed
- Type systems shape valid programs
- Frameworks shape architecture
- Patterns shape recurring solutions
- Coding standards shape taste
- Tests shape behaviour
- Stories shape intent
- PRs shape change boundaries
- CI shapes admissibility
- Retros shape future practice

LLMs and AI agents are trained on the artefacts of software engineering, so they naturally reflect the strengths and weaknesses of those artefacts while exposing context that is missing, too implicit, or weakly encoded.

The point is, harness engineering isn't anything new. In fact, if it didn't expose the real issue, I'd go so far as to call it completely redundant.

## Why Harness Engineering Feels New

Harness engineering feels new because LLMs are less forgiving of vague or implicit shapes. They struggle when context-related knowledge is invisible or entirely lost:

- People's heads
- Slack threads
- Past meetings
- Review comments
- Inconsistent codebases
- Underspecified stories
- Weak tests

Humans can recover some of that context from memory, conversation, or inference. Agents need it encoded.

Humans are implicit, computers are explicit, and AI's somewhere in between.

Also, the human brain generally wants to conserve energy. Computers, given enough resources, will compute endlessly.

This means, even with the best intentions, things slip through the cracks:

- A story says what to build, but not enough about acceptable behaviour
- A style guide describes principles, but examples are missing
- Review feedback corrects a mistake, but the correction is not turned into a reusable constraint
- Tests verify happy paths, but not the behaviour that actually matters
- Architecture is understood socially, but not visible locally in the repo

The truth is, even before LLMs came along, imprecision has always been a major issue that produces a massive amount of waste.

## The Real Work Is Precision

We don't need to reinvent the wheel. We need practices that reward precision instead of relying on shared inference, social memory, and unstated taste.

The work is to make shapes more explicit:

- Make implicit conventions explicit
- Move knowledge closer to the code
- Turn repeated feedback into tests, docs, templates, lint rules, or examples
- Prefer executable constraints over prose
- Define work in smaller, more atomic, clearer units
- Tighten feedback loops

So the next practical question is not "how do we build a better harness?" It is "how can we be more precise?"

That means asking what context we're missing, where we're being too implicit, which feedback loops we should establish, and how we can reduce corrections or waste.

Harness engineering is useful as a lens, but misleading as the destination. The real work is extending software engineering practice so its shapes are legible to both humans and agents.
